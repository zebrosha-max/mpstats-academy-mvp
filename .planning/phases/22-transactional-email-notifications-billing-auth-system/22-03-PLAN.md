   - Import `prisma` from `@mpstats/db/client`, `cq` from `@/lib/carrotquest/client`, `isFeatureEnabled` check

   **Expiry reminders (CANCELLED subscriptions expiring in 3 days):**
   - Query: `prisma.subscription.findMany({ where: { status: 'CANCELLED', currentPeriodEnd: { gte: threeDaysFromNow_start, lte: threeDaysFromNow_end } }, include: { user: true, plan: { include: { course: true } } } })`
   - Where `threeDaysFromNow_start` = start of day +3 days, `threeDaysFromNow_end` = end of day +3 days
   - For each, track event `$subscription_expiring` with vars: name, course_name, access_until, renew_url (`${SITE_URL}/pricing`)

   **Inactivity chain:**
   - Query users with no `LessonProgress` update in 7/14/30 days:
     ```sql
     SELECT u.id, u."fullName", u.email, MAX(lp."updatedAt") as last_activity
     FROM "UserProfile" u
     LEFT JOIN "LessonProgress" lp ON lp."userId" = u.id
     GROUP BY u.id
     HAVING MAX(lp."updatedAt") < NOW() - INTERVAL '7 days'
     ```
   - Use Prisma raw query or aggregation. Practical approach:
     - Get all users with their latest LessonProgress updatedAt
     - Filter by 7/14/30 day thresholds
     - Track appropriate event (`$inactive_7d`, `$inactive_14d`, `$inactive_30d`)
   - IMPORTANT: Need deduplication -- don't send the same inactivity email twice. Options:
     a) CQ handles dedup via triggered message settings (send once per event)
     b) Track last notification sent in a simple table or use CQ user props
     - Simplest: rely on CQ triggered message "send once" setting. Document this in EMAIL-SPEC.md.
   - Return JSON with counts: `{ expiring: N, inactive_7d: N, inactive_14d: N, inactive_30d: N }`
