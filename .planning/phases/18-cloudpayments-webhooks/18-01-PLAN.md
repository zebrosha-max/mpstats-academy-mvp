<task type="auto">
  <name>Task 2: Webhook route handler with idempotency and audit logging</name>
  <files>apps/web/src/app/api/webhooks/cloudpayments/route.ts</files>
  <action>
Create Next.js API route `apps/web/src/app/api/webhooks/cloudpayments/route.ts`:

1. Export `const dynamic = 'force-dynamic'`

2. Export async POST handler:
   a. Read raw body as text (`await request.text()`) — needed for HMAC verification
   b. Get `Content-HMAC` header from request
   c. Call `verifyCloudPaymentsHmac(rawBody, hmacHeader)` — if false, log warning with IP and return `NextResponse.json({ code: 13 }, { status: 403 })`
   d. Parse body as CloudPaymentsWebhookPayload (JSON.parse(rawBody))
   e. Determine event type from request URL path or a custom header. CloudPayments sends webhooks to different endpoints per event type. Handle this by checking the last path segment OR by using a single route with a query param. **Recommended approach**: use a single route, determine event type from `Status` and `OperationType` fields in payload:
      - OperationType="Payment" + Status="Completed" -> "pay"
      - OperationType="Payment" + Status="Declined" -> "fail"
      - OperationType="Refund" -> "refund"
      - If payload has `Token` and `Interval` fields -> "recurrent" (recurrent subscription notification)
      - Otherwise fallback to logging as "unknown"

      **Alternative (simpler, per CloudPayments docs):** CloudPayments sends to separate URLs per notification type: /check, /pay, /fail, /recurrent, /cancel. Since we control webhook URL configuration in CloudPayments dashboard, use a SINGLE catch-all route and pass event type via query parameter: `/api/webhooks/cloudpayments?type=pay`. Parse `type` from `request.url` searchParams.

   f. **Idempotency check**: `prisma.payment.findUnique({ where: { cloudPaymentsTxId: String(payload.TransactionId) } })`
      - If found AND status is COMPLETED -> log as duplicate, still create PaymentEvent for audit, return `{ code: 0 }`
      - If found AND status is PENDING -> proceed with update (pay/fail will finalize it)

   g. **Payment upsert**: Use `prisma.payment.upsert()`:
      - where: `{ cloudPaymentsTxId: String(payload.TransactionId) }`
      - create: new Payment with subscriptionId from `payload.InvoiceId`, amount from `payload.Amount`, cloudPaymentsTxId, status based on event type (pay -> COMPLETED, fail -> FAILED, check -> PENDING)
      - update: status and paidAt fields

   h. **PaymentEvent create**: Always create PaymentEvent record:
      ```typescript
      await prisma.paymentEvent.create({
        data: {
          paymentId: payment.id,
          type: eventType,
          payload: JSON.parse(rawBody) // full payload as Json
        }
      })
      ```

   i. **Return**: `NextResponse.json({ code: 0 })` for success. CloudPayments retries if it gets anything other than `{ "code": 0 }`.

3. Error handling:
   - Wrap everything in try/catch
   - On JSON parse error -> return `{ code: 13 }` (invalid payload)
   - On Prisma error -> log error, return `{ code: 0 }` (accept webhook to prevent retries, log for investigation)
   - On unknown error -> log, return `{ code: 0 }`

4. Import `prisma` from `@mpstats/db/client` (same pattern as Yandex OAuth callback route).

**Important:** Subscription status transitions (ACTIVE, PAST_DUE, CANCELLED) are handled in Plan 02. This task focuses on Payment creation, idempotency, and audit logging only. The route will be extended in Plan 02 to call a subscription service.
