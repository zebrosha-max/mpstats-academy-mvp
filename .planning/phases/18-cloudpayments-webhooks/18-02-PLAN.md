  <action>
Create `apps/web/src/lib/cloudpayments/subscription-service.ts` with the following exported functions:

1. **`handlePaymentSuccess(subscriptionId: string, payment: { id: string; amount: number }): Promise<void>`**
   - Find subscription by id with its plan (include: { plan: true })
   - If not found, log error and return (non-fatal — Payment was already saved in Plan 01)
   - Set status = ACTIVE
   - Set currentPeriodStart = now
   - Set currentPeriodEnd = now + plan.intervalDays days
   - Update via `prisma.subscription.update()`

2. **`handlePaymentFailure(subscriptionId: string): Promise<void>`**
   - Find subscription by id
   - If not found, log and return
   - If status is ACTIVE or PAST_DUE -> set status = PAST_DUE
   - If status is CANCELLED or EXPIRED -> no change (already terminal-ish)
   - CloudPayments has built-in retry logic for failed recurrents. Grace period: subscription stays PAST_DUE until CloudPayments gives up retrying (typically 3 attempts over several days). When all retries exhausted, CloudPayments sends cancel event.

3. **`handleCancellation(subscriptionId: string): Promise<void>`**
   - Find subscription by id
   - If not found, log and return
   - Set status = CANCELLED
   - Set cancelledAt = now
   - **DO NOT set to EXPIRED** — user retains access until currentPeriodEnd (per context decision). The EXPIRED transition happens at access-check time in Phase 20 (if currentPeriodEnd < now && status == CANCELLED -> treat as EXPIRED)

4. **`handleRecurrentPayment(subscriptionId: string, payment: { id: string; amount: number }): Promise<void>`**
   - Same logic as handlePaymentSuccess but specifically for recurring billing cycle
   - Find subscription with plan
   - Set status = ACTIVE (in case it was PAST_DUE from a failed attempt)
   - Extend currentPeriodEnd by plan.intervalDays from the CURRENT currentPeriodEnd (not from now — to avoid gaps or overlaps in billing periods)
   - Update currentPeriodStart = old currentPeriodEnd

5. **`handleCheck(accountId: string, invoiceId: string): Promise<boolean>`**
   - Check webhook is called BEFORE payment. Return true if we want to accept the payment.
   - Verify accountId (userId) exists in UserProfile
   - Verify invoiceId (subscriptionId) exists and belongs to accountId
   - If validation fails, return false (CloudPayments will decline the payment)
   - If OK, return true

Import `prisma` from `@mpstats/db/client`.
All functions wrap DB calls in try/catch and log errors with `console.error` (no throw — webhook handler must always return {code: 0}).
