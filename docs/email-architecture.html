01

Как реально работает доставка писем

Когда юзер регистрируется, сбрасывает пароль или меняет email, Supabase Auth не идёт в SMTP. Вместо этого он стреляет HTTPS‑webhook'ом на наш сервер, а мы уже посылаем событие в CarrotQuest, который и отправляет письмо.

02

Send-Email Hook — перехватчик

Один Next.js route принимает все auth-email события от Supabase и маршрутизирует их в CQ. Конфиг хука прописан на стороне Supabase, секрет верифицируется по Standard Webhooks.

📑 Какие события приходят
email_action_typeВ CQ как
signup event pa_doi + prop pa_doi=<url>
recovery event pa_password_reset + prop pa_password_link
email_change event pa_email_change + prop pa_confirm_url
magic_link не используется в продукте
invite не используется
reauthentication не используется
↡ Что делает webhook
  1. Принимает POST с HMAC-подписью webhook-id/webhook-timestamp/webhook-signature
  2. Верифицирует подпись через SUPABASE_HOOK_SECRET (формат v1,whsec_BASE64)
  3. Парсит payload — достаёт user.id, email_action_type, token_hash, redirect_to
  4. Собирает confirmUrl = supabase.co/auth/v1/verify?token=...&type=...&redirect_to=...
  5. Кидает cq.setUserProps() с линком и cq.trackEvent() с именем события
  6. Возвращает 200 — всегда, даже на ошибке (чтобы не ломать auth flow Supabase)
03

Транзакционный канал — CQ event API

Помимо auth-писем через webhook, есть прямые вызовы CQ из нашего кода — платежи, истечения, неактивность. Все идут по тому же паттерну: setUserProps+trackEvent, CQ дальше срабатывает по настроенным automation rules.

04

Что реально пошло не так с bakaresh@yandex.ru

Изначально я грешил на rate-limit Resend — неверно. Поднял auth-конфиг Supabase, там hook_send_email_enabled=true, Resend dashboard показывает 0 писем за месяц. Реальная причина — silent-drop trap в цепочке webhook→CQ.

Корень проблемы: CQ-клиент apps/web/src/lib/carrotquest/client.ts ловил все ошибки fetch'а и !response.ok в console.error, не пробрасывая их наружу. Webhook видел успешный return, отдавал Supabase 200, юзер не получал письмо. В логах оставалась только строка в stdout Docker-контейнера — через рестарт ротировалась и терялась.

  1. Юзер регистрируется → Supabase POST'ит webhook'у нашему серверу
  2. Handler зовёт cq.trackEvent('pa_doi')
  3. CQ отдаёт 4xx/5xx (rate-limit, glitch, suppressed lead, что-угодно)
  4. Старый код: console.error('CQ error 500') — и всё. Никто наверх не узнал
  5. Webhook возвращает 200 — Supabase помечает confirmation_sent_at=now()
  6. Юзер ничего не получает. В БД всё выглядит нормально
  7. Никаких алертов, никакого Sentry — письмо тихо умерло
05

Что зафиксили — делаем silent‑drop невозможным

Сегодня (27.04) задеплоили изменения, после которых любая такая ошибка станет видна в Sentry в течение секунд. Не увеличивает нагрузку на CQ, не ломает auth-flow.

🔧 Три точки наблюдения
ГдеЧто меняетсяЭффект
lib/carrotquest/client.ts request() теперь throw'ит CQApiError на !response.ok и CQNetworkError на fetch‑exception. Добавлены breadcrumbs Любая CQ‑ошибка пробрасывается до Sentry-catch
lib/carrotquest/emails.ts Все 6 transactional email функций теперь зовут Sentry.captureException с tags area=carrotquest-email и stage=<функция> В Sentry видно какой именно email сломался для какого юзера
api/webhooks/supabase-email/route.ts Перед обработкой выставляются Sentry.setTags({email_action_type}) и Sentry.setUser({id, email}). Catch теперь ловит CQ-ошибки и помечает их area=supabase-email-hook В Sentry можно фильтровать по типу события и юзеру
✅
Поведение auth-flow не изменилось: webhook всё ещё возвращает 200 на ошибки, чтобы Supabase не ломал регистрацию. Но теперь Sentry знает что именно сломалось, и через alert rules команда может реагировать в течение минут.
📋
Что ещё стоит сделать (не в этом релизе): retry с backoff в CQ-клиенте (3 попытки на 5xx), alert rule в Sentry на area=carrotquest-email с уведомлением в Telegram, dashboard в CQ с deliverability-метрикой по pa_doi.

🔎 CarrotQuest dashboard

Лиды → найти по email → вкладка «События» → искать pa_doi или pa_password_reset.

Если события нет вообще — webhook не дошёл до CQ (смотреть Sentry).

Если событие есть, но письма нет — проблема в automation rule на стороне CQ (suppression, фильтр сегмента, отвалившийся шаблон). Это уже вотчина CRM-команды.
