# Entry Flow Redesign — Развилка входа на платформу

**Date:** 2026-05-18
**Source:** Vlad Tokarev (CPO) — «План доработки платформы», НАПРАВЛЕНИЕ 03 + ПРОТОТИП 03 (слайды 11–14)
**Status:** Design approved, ready for phase planning

## Problem

Сегодня платформа жёстко завязана на диагностику:

- **Жёсткий гейт на странице урока.** `apps/web/src/app/(main)/learn/[id]/page.tsx:641-645` — если диагностика не пройдена (`hasCompletedDiagnostic === false`), весь контент урока (плеер, конспект) подменяется блокирующим `DiagnosticGateBanner` с кнопкой «Начать диагностику». Посмотреть нельзя ни один урок.
- **Нет квалификации.** Платформа ничего не знает о пользователе на входе — ни маркетплейс, ни опыт, ни цели. Диагностика и контент не сегментированы под аудиторию.
- **Дашборд встречает пустотой.** Первый экран — 4 нулевых счётчика + две CTA-карточки.

Последствия (из аудита White Paper): опытный селлер не хочет проходить тест и упирается в стену; новичок проходит тест без сегментации и получает нерелевантный план. Оба теряются на activation.

## Goal

Сделать вход на платформу комфортным и дать пользователю **выбор пути** с самого начала:

1. Мягкий онбординг-визард вместо «диагностика в лоб».
2. Сбор квалификации (маркетплейсы / опыт / цели) — основа для персонализации.
3. Равноценная развилка: диагностика **или** сразу в каталог уроков.
4. Снять жёсткий гейт диагностики — уроки доступны в рамках подписки без диагностики.

**Метрики:** Activation rate ↑ (целевое ≥40%), drop на онбординге ↓.

## Non-Goals (вне скоупа)

- Редизайн раздела «Обучение» / библиотеки / плейбуков — отдельная следующая задача (раскатка на staging).
- Реальный AI-ответ на экране 1 («Вариант 3») — отложено до стабилизации поиска (см. memory `entry-flow-screen1-v3-future`).
- Переделка счётчиков дашборда / комьюнити-блок — это НАПРАВЛЕНИЕ 06, отдельная фаза.
- Диагностика как лид-магнит за контуром продукта — будущее.
- Изменение подписочного гейта (`lesson.locked` / `LockOverlay`) — остаётся как есть.

## Solution Overview

Новый одноразовый онбординг-визард `/welcome` между регистрацией и продуктом:

```
Регистрация (DOI / Yandex OAuth)
        │
        ▼
   /welcome  (визард, показывается один раз)
   ├─ Шаг 1: Намерение + цели
   ├─ Шаг 2: Маркетплейсы
   ├─ Шаг 3: Опыт
   └─ Развилка
        ├─ «Пройти диагностику»  → /diagnostic
        └─ «Перейти в обучение»  → /learn
        │
        ▼
   Последующие входы → /dashboard (как сейчас)
```

## Screens

### Шаг 1 — Намерение + цели

Помощник приветствует пользователя по имени. Вопрос: «Зачем вы пришли в Академию?»

- **Свободное текстовое поле** (необязательно) — пользователь описывает задачу своими словами.
- **7 чипов-целей, multi-select:** Увеличить продажи · Снизить расходы на рекламу / настроить продвижение · Улучшить карточки товара · Разобраться в аналитике и нишах · Навести порядок в операциях и логистике · Финансы и юнит-экономика · Выйти на новый маркетплейс.

На сабмит — **честная переформулировка одной строкой** («Поняли — хотите …»), затем переход к шагу 2. Переформулировка — простой клиентский шаблон (эхо выбранных чипов или введённого текста), **без вызова LLM** в этой фазе. Платформа **не имитирует** решение задачи (это отличие от прототипа Влада, где первый экран выглядел как AI-чат, но ответа не давал — bait-and-switch, который бьёт по доверию).

### Шаг 2 — Маркетплейсы

Вопрос: «Выберите маркетплейсы, на которых работаете». **Multi-select:** Wildberries · Ozon · Яндекс Маркет · AliExpress · Мегамаркет · Свой интернет-магазин · Другое.

### Шаг 3 — Опыт

Вопрос об уровне опыта. **Single-select:**
- Только присматриваюсь — ещё не продаю
- Новичок — продаю недавно, небольшой оборот
- Есть стабильные продажи
- Опытный селлер — уверенно масштабируюсь

### Развилка (финальный экран)

Текст: «Мы готовы помочь вам расти». Две **равноценные** карточки:
- 🟪 **Пройти диагностику** — «15 вопросов, подберём уроки под проблемные зоны» → `/diagnostic`.
- 🟩 **Перейти в обучение** — «Выберите темы, которые интересны сейчас» → `/learn` (каталог в текущем состоянии).

Внизу честная строка: «В любой момент можно пройти диагностику или изменить путь в настройках профиля».

Клик по любой карточке → мутация `onboarding.complete` (сохраняет квалификацию + ставит `onboardingCompletedAt`) → навигация на выбранный путь.

Степпер показывает 3 шага (шаг 1 = первый). Развилка не нумеруется.

## Data Model

Новые поля на модели `UserProfile` (`packages/db/prisma/schema.prisma`). Паттерн уже есть — `toursCompleted String[]`.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `onboardingCompletedAt` | `DateTime?` | `null` = показать визард. Ставится при прохождении развилки. |
| `marketplaces` | `String[]` | Выбранные маркетплейсы (шаг 2). |
| `experienceLevel` | `String?` | Уровень опыта (шаг 3). |
| `goals` | `String[]` | Выбранные цели-чипы (шаг 1). |
| `goalText` | `String?` | Свободный текст с шага 1. |

Значения кодируются стабильными строковыми ключами (например `WB`, `OZON`, `OWN_SHOP`; `PROSPECTING`, `BEGINNER`, `STABLE`, `ADVANCED`; `SALES`, `ADS`, `CONTENT`, `ANALYTICS`, `OPERATIONS`, `FINANCE`, `NEW_MARKETPLACE`). Выбор между Prisma enum и `String[]` — на этапе планирования; `String[]` повторяет существующий паттерн `toursCompleted`.

Schema migration выполняется на prod **до** rebuild (recurring lesson — Phase 28).

## Routing & Guard

- **Новый роут `/welcome`** — standalone-роут **вне** группы `(main)`, с минимальным полноэкранным layout (без основного сайдбара/навигации). Клиентский визард: состояние шагов держится на клиенте; одна tRPC-мутация `onboarding.complete` в конце сохраняет всё. Если пользователь бросил визард на полпути — клиентское состояние теряется, при следующем входе визард начинается заново (для флоу на ~30 секунд пошаговый серверный save не нужен).
- **Гард** в server-component layout группы `(main)`: если пользователь авторизован и `onboardingCompletedAt == null` → редирект на `/welcome`. Гард в layout, а не в edge-middleware, потому что нужен запрос к БД (Prisma в edge-middleware ненадёжен). Поскольку `/welcome` лежит вне `(main)`, гард на него не распространяется — отдельного исключения по пути не нужно. Этот гард автоматически ловит и текущих ~200 пользователей при следующем входе — у всех `onboardingCompletedAt` пустой.
- `/auth/confirm` и `/auth/callback` продолжают редиректить на `/dashboard`; гард `(main)`-layout перехватит и уведёт на `/welcome` при необходимости.

## Diagnostic De-gating

Снять жёсткий гейт диагностики (отдельный от подписочного):

- `apps/web/src/app/(main)/learn/[id]/page.tsx:641-645` — убрать ветку `hasDiagnostic === false → <DiagnosticGateBanner/>`. Урок рендерится для всех; остаётся только подписочная ветка `lesson.locked → <LockOverlay/>`.
- `DiagnosticGateBanner` → переделать из блокирующего баннера в **ненавязчивую закрываемую карточку-хинт** над плеером: «Пройди диагностику — соберём персональный трек». Показывается только при `hasCompletedDiagnostic === false`; после закрытия не возвращается (флаг в `localStorage`). Видео при этом всегда доступно.
- Серверная процедура `learning.getLesson` урок отдаёт всегда — серверного гейта нет, правка чисто клиентская.
- Каталог `/learn` — там баннер-подсказка про трек уже не блокирующий, трогать не обязательно.

Итог: без диагностики пользователь смотрит **все уроки в рамках своей подписки**. Подписочный гейт `lesson.locked` / `LockOverlay` не затрагивается.

## tRPC API

Новый роутер `onboarding` в `packages/api/src/routers/`:
- `getState` (query) — возвращает `onboardingCompletedAt` и сохранённую квалификацию (для гарда и страницы профиля).
- `complete` (mutation) — принимает `{ marketplaces, experienceLevel, goals, goalText }`, сохраняет на `UserProfile`, ставит `onboardingCompletedAt = now()`.

Квалификация редактируется позже на странице `/profile` (поля переиспользуют те же значения).

## Integration & Handoff

Собранные `marketplaces` + `goals` — это вход для **НАПРАВЛЕНИЯ 01** (унификация диагностики, «релевантность вопросов под аудиторию WB»). Эта фаза только **складывает** данные; потребление — задача направления 01.

## Out-of-Scope Future (recorded)

«Вариант 3» экрана 1 — превратить экран намерения в реальный 1-turn AI-ассистент (отвечает на задачу подобранными уроками/плейбуками) — отложен до стабилизации каталог-поиска. Зафиксировано в memory `entry-flow-screen1-v3-future`. Триггер пересмотра — работы по качеству поиска (Phase 56 retrieval).

## Testing

- Unit: `onboarding.complete` сохраняет поля и ставит таймстемп; `getState` возвращает корректное состояние.
- E2E: новый пользователь → `/welcome` → 3 шага → развилка → обе ветки (`/diagnostic` и `/learn`); визард не показывается повторно после прохождения.
- E2E: пользователь без диагностики открывает урок → видео доступно, хинт показан и закрывается; подписочный `LockOverlay` для платного урока по-прежнему блокирует.
- Регрессия: существующий пользователь с `onboardingCompletedAt == null` видит визард один раз при следующем входе.
