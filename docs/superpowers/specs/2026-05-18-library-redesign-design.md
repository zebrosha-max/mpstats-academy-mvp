# Library Redesign — Каталог `/learn` на джобах

**Дата:** 2026-05-18
**Статус:** Design — утверждён владельцем, готов к планированию
**Фаза:** Phase 57 (v1.8 / следующая). Follow-up — Phase 58 (миграция диагностики на джобы).

---

## 1. Контекст и проблема

`/learn` сегодня — это тоггл «Мой трек / Все курсы», AI-поиск с чипами-примерами, панель из 7 фильтров и компонент `LibrarySection` (Phase 46), скрытый на проде за флагом `NEXT_PUBLIC_SHOW_LIBRARY`. «Phase 47» (hub-layout) была зафиксирована как архитектурная заметка, но никогда не реализована.

Боли:

- **Аккордеон громоздкий.** В «Все курсы» развернуть несколько курсов = неповоротливая простыня.
- **Дублирование.** Одни уроки в «Все курсы» и в «Библиотеке» — непонятно, чем пользоваться.
- **Нет ощущения продукта.** Платформа воспринимается как набор курсов, а не как инструмент решения конкретных задач селлера.
- **Две несвязанные системы классификации.** Диагностика рекомендует уроки по грубому полю `skillCategory` (5 категорий). 32-блочный `skillBlocks` (Phase 46 skill-mapping) кормит только `LibrarySection`. Они не разговаривают — рекомендация после диагностики крупная и неточная («слаб в рекламе → 5 случайных уроков MARKETING»).

**Цель:** пересобрать каталог вокруг **джоб** (jobs-to-be-done) — задач селлера. Платформа становится «бери набор и решай задачу», а не «свалка курсов». Тот же контент, новая компоновка. Раскатать на staging.

## 2. Решение — обзор

Каталог `/learn` строится вокруг новой сущности **Джоба** — курируемого упорядоченного набора уроков под одну задачу селлера. Джоба — независимый слой: course-agnostic, many-to-many с уроками, выводится из **содержания** уроков (транскриптов), валидируется контент-командой.

Работа разбита на два потока, связанных одним контрактом (схема `Job`):

- **Workstream A — Job Mapping.** Контент-анализ уроков → черновая таксономия джоб → JOB-PROPOSAL для валидации контент-командой.
- **Workstream B — Library UI.** Редизайн `/learn` (5 экранов) против схемы `Job`. Строится на провизорных данных параллельно валидации A, затем подмена на утверждённые джобы → staging.

Джоба проектируется сразу как **мост к диагностике** (несёт `axes` + `skillBlocks`), но сам перевод диагностики на джобы — отдельная Phase 58, в этот спек не входит.

## 3. Модель сущностей

| Сущность | Что это | Статус |
|---|---|---|
| **Урок** (`Lesson`) | атом — видео + транскрипт | есть, не меняем |
| **Джоба** (`Job`) | курируемый упорядоченный набор уроков под задачу селлера; course-agnostic; M2M с уроками | **новая** |
| **Курс** (`Course`) | бандл уроков; единица биллинга (тариф COURSE) | есть, не меняем |
| **Скилл-блок** | сквозной тег (32 блока, поле `Lesson.skillBlocks`) | есть; роль — фильтр/поиск + мост к диагностике |
| **Трек** (`LearningPath`) | личный план юзера; секции из джоб + ручные уроки | есть; станет job-aware |

Иерархия: **Курс ⊃ Урок** и **Джоба ⊃ Урок** — два независимых способа группировки одних и тех же уроков. Скилл-блок — ортогональный тег. Джоба ≠ модуль: триал показал, что 58% джоб кросс-курсовые (см. §10).

**Имя для пользователя** («джоба / набор / плейбук / сценарий») — решение по UI-копирайту, отложено. В коде/схеме — `Job`.

## 4. Схема `Job`

Новая модель Prisma + join-таблица для упорядоченной M2M связи с уроками.

```prisma
model Job {
  id          String   @id @default(cuid())
  slug        String   @unique          // URL: /learn/job/<slug>
  title       String                    // глагол + результат: "Посчитать юнит-экономику товара"
  description String                    // 1-2 предложения
  outcomes    Json                      // string[] — "что ты сможешь после"
  axes        Json                      // canonical 5: ["FINANCE","ANALYTICS"] — мост к диагностике
  skillBlocks Json                      // 32-блочные теги — точный мост к диагностике
  marketplace JobMarketplace @default(WB) // WB | OZON | BOTH
  displayOrder Int      @default(0)      // порядок внутри оси в каталоге
  isPublished Boolean  @default(false)   // черновик / опубликована
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  lessons     JobLesson[]
}

model JobLesson {
  jobId    String
  lessonId String
  order    Int                          // порядок урока внутри джобы
  job      Job    @relation(fields: [jobId], references: [id], onDelete: Cascade)
  lesson   Lesson @relation(fields: [lessonId], references: [id], onDelete: Cascade)
  @@id([jobId, lessonId])
  @@index([lessonId])
}

enum JobMarketplace { WB OZON BOTH }
```

- `axes` хранит **канонические 5 категорий** — это и есть данные моста: диагностика меряет 5 категорий, и джоба тегается ими.
- `skillBlocks` — для точного матча после Phase 58.
- `marketplace` — WB и Ozon разные ветки; `BOTH` для кросс-платформенных джоб (юнит-экономика, P&L).
- `level` (новичок/опытный) — **не вводим** на этом этапе (отложенное решение, §11).
- Целевое число джоб — **~30-40**, плоско, без иерархии «джоба → под-джоба».

`schema.prisma` в `packages/db/` — единственный источник истины для DDL прод-БД (см. PROD DATABASE SAFETY). Миграция — через стандартный flow репозитория MAAL.

## 5. Workstream A — Job Mapping

Пайплайн `scripts/job-mapping/` — по паттерну существующего `scripts/skill-mapping/`, но классификация по **содержанию** (транскрипты `content_chunk`), не по названиям.

**Этапы:**

1. **Discover** — по каждому уроку LLM читает транскрипт и формулирует 1-3 задачи селлера + `axes` (строго из 5 категорий) + `skillBlocks`.
2. **Cluster** — консолидация в ~30-40 джоб. **Marketplace-aware:** WB-only и Ozon-only уроки не смешиваются в одну джобу; кросс-платформенные джобы метятся `BOTH`. Оси кластера **жёстко ограничены** 5 канонами (триал показал — иначе LLM выдумывает свои, мост ломается).
3. **JOB-PROPOSAL** — человекочитаемый документ (джоба → описание → уроки по порядку → axes/skillBlocks → marketplace) → контент-команде на валидацию и правку.

**Вход:** только уроки `isHidden = false` в видимых курсах. Курс `06_express` исключён целиком (см. §9).

**Модель:** `openai/gpt-4.1-mini` (выбор проекта по качеству — см. Key Decisions). Стоимость полного прогона ~$3-5.

**Выход для Workstream B:** валидированный JOB-PROPOSAL → seed-скрипт наполняет таблицы `Job` / `JobLesson`.

## 6. Workstream B — Library UI

Пять экранов (HTML-прототип каталога согласован с владельцем в брейншторм-сессии; прототип — throwaway, в репозиторий не входит):

| Экран | URL | Содержание |
|---|---|---|
| **Каталог — по задачам** | `/learn` (default) | джобы, сгруппированные по 5 осям; карточка джобы (название, описание, N уроков, прогресс, метка marketplace/«рекомендовано»); тонкие фильтры |
| **Каталог — по курсам** | `/learn` (линза) | тот же контент по курсам — эволюция сегодняшнего «Все курсы»; сохраняет тариф COURSE |
| **Мой трек** | `/learn/track` | полный вид `LearningPath`: секции из джоб + «Мои уроки»; «Продолжить», «Перестроить по диагностике» |
| **Страница джобы** | `/learn/job/<slug>` | заголовок, «что сможешь после», уроки по порядку, «Продолжить джобу» / «+ В трек» |
| **Поиск** | `/learn?q=...` | плоский список уроков-результатов + полная панель фильтров |

**Ключевые решения по UI:**

- `/learn` по умолчанию открывается линзой **«по задачам»**; «по курсам» — в один клик (тоггл-линза).
- **Маркетплейс** — переключатель `Wildberries / Ozon` вверху страницы (две ветки контента); джобы `BOTH` видны в обеих.
- **Трек** получает собственный URL `/learn/track` (не тоггл на `/learn`). На каталоге — компактный баннер трека с двумя действиями: «Открыть трек» (полный вид) и «Продолжить» (прыжок в следующий незавершённый урок).
- **Аккордеон** убирается из каталога; клик по джобе → страница джобы. Лёгкая раскрывашка остаётся только в линзе «по курсам» и в треке.
- `LibrarySection` (Phase 46) и флаг `NEXT_PUBLIC_SHOW_LIBRARY` — заменяются новым каталогом.

**Данные:** UI строится против схемы `Job`; до готовности Workstream A — на провизорном seed (несколько джоб руками), затем подмена на валидированные. → staging.

## 7. Фильтры — раскладка

Сегодняшние 7 фильтров (`FilterPanel`) спроектированы под плоский список уроков. В job-каталоге:

| Фильтр сегодня | Решение |
|---|---|
| Категория (5 осей) | → **группировка каталога** (5 секций) + быстрые якоря; как фильтр не нужен |
| Маркетплейс | → **верхний переключатель WB/Ozon** |
| Курс | только в линзе «по курсам» |
| Статус (прогресс) | **остаётся** на каталоге — на уровне джобы |
| Сложность | урок-уровень → только на экране «Поиск» |
| Длительность | урок-уровень → только на экране «Поиск» |
| Топики | урок-уровень → только на экране «Поиск» |

Итог: каталог «по задачам» — тонкие фильтры (статус). Полный набор урок-фильтров живёт на экране «Поиск», где контент — плоский список уроков. Логика и компонент поиска по сути не переписываются — переезжают на свой экран. Чипы-примеры под строкой поиска остаются.

## 8. Мост к диагностике

Сегодня диагностика рекомендует уроки по `skillCategory` (5 категорий). Джоба несёт `axes` (5 категорий) + `skillBlocks` (32) — это «разъём», в который диагностика сможет воткнуться: рекомендовать **джобы целиком**, а не россыпь уроков.

В этой фазе (57) мост — только **данные и контракт**: схема `Job` несёт нужные поля; определяется интерфейс выборки джоб по слабым навыкам (напр. `getJobsForSkillGaps()`), возможно заглушкой. `diagnostic.ts` **не меняется**.

Риск, заложенный в дизайн: данные моста богаче, чем измеряет диагностика (5 категорий vs 32 блока) — поэтому джоба тегается **и** канонами-5 (грубый матч работает сразу), **и** скилл-блоками (точный матч — после Phase 58).

**Phase 58 (follow-up, отдельный спек):** реальный перевод — `diagnostic.ts` рекомендует джобы, `getRecommendedPath` и «Мой трек» работают на джобах.

## 9. Предусловия и ограничения

- **Express исключается.** Курс `06_express` — `isHidden=true`, но его 64 урока `isHidden=false`. Контент дублирует основные курсы (подтверждено совпадением названий: `02_ads`+`06_express`, `03_ai`+`06_express`, `01_analytics`+`06_express`). Job-mapping работает только по `isHidden=false` урокам **в видимых курсах** → express не попадает. Решение «удалять ли express совсем» — вызов контент-команде, не блокирует.
- **Dedup — предусловие.** `01_analytics` содержит 42 скрытых урока и дубли названий внутри курса («Работа с фокусными товарами» ×5, «Стратегическое планирование» ×4). Перед полным прогоном Workstream A — убедиться, что dedup чистый, иначе джобы наберут фантомных дублей.
- **Прод-БД.** Миграция `Job`/`JobLesson` — только из репозитория MAAL, по pre-flight checklist (PROD DATABASE SAFETY в `MAAL/CLAUDE.md`).
- **Маркетплейс-разделение** — WB-only и Ozon-only уроки не смешиваются в одной джобе.

## 10. Доказательная база — пробный прогон

Триал (`scripts/job-mapping/trial-run.ts`, `gpt-4.1-mini`, 35 уроков по всем курсам):

- **12 джоб, 7 кросс-курсовых (58%)** — подтверждает: джоба шире модуля/курса, «модуль = джоба» не работает.
- Качество джоб содержательное («Подготовиться к сезону Q4», «Найти недоработки в карточке и поднять продажи»).
- 2 сироты из 35 — честные edge-cases.
- Вскрыло 2 вещи, учтённые в дизайне: оси кластера надо жёстко ограничивать 5 канонами; целевое число джоб укрупнять до ~30-40.

Артефакты: `scripts/job-mapping/results/trial-run-digest.md`.

## 11. Out of scope

- Перевод диагностики на джобы (Phase 58).
- Удаление курса `06_express` (решение контент-команды).
- Страница урока, видео-плеер, прогресс, paywall/`LockOverlay` — не трогаются.
- Переписывание логики поиска/семантики — поиск только переезжает на свой экран.
- AI-ответ на экране входа, entry-flow (Phase 56 — отдельный спек).

## 12. Открытые вопросы / отложенные решения

- **Имя сущности для пользователя** — «джоба / набор / плейбук / сценарий». Решается на этапе UI-копирайта.
- **`Job.level`** (новичок/опытный) — не вводим сейчас; вернуться, если появится явная потребность.
- **Фильтр длительности джоб** («короткие / длинные») — nice-to-have, не в скоупе MVP.
- **Плейсхолдер строки поиска** — поправить текст (мелочь, отдельно).
- **Объяснение пользователю состава осей** — UX-копирайт: что входит в большую ось.
