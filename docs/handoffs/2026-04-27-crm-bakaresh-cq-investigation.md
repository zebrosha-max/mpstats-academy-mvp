# Инструкция для CRM-команды — расследование пропавшего DOI-письма

**Дата:** 2026-04-27
**Кейс:** `bakaresh@yandex.ru` — зарегался 23.04 в 14:14 МСК, до 27.04 утра не получил письмо подтверждения регистрации
**Что нужно:** проверить в CarrotQuest, что произошло, чтобы понять системную причину

---

## Контекст в одной фразе

Все наши auth-письма (подтверждение регистрации, сброс пароля, смена email) идут через CarrotQuest. Платформа стреляет событие в CQ, automation rule в CQ отправляет письмо. Один юзер не получил DOI-письмо 23.04 — нужно понять, проблема в нашей платформе или в CQ.

## Что проверить — пошагово

### Шаг 1. Найти лид в CarrotQuest

1. Открыть CQ-дашборд
2. Раздел **«Контакты»** / **«Лиды»**
3. Поиск по email: `bakaresh@yandex.ru`
4. Если лид есть — открыть карточку

**Что записать:**
- ID лида
- Дата создания (должна быть около 23.04 14:14 МСК)
- Заполненные свойства (`pa_name`, `pa_phone`, `$email`, `$name` — должны быть)

**Если лида нет** — сразу пинг разработчику. Это значит webhook от Supabase не дошёл до CQ ни разу — баг на нашей стороне.

### Шаг 2. Проверить события лида

1. В карточке лида открыть вкладку **«События»** (или **«Activity»** / **«Журнал»** — зависит от интерфейса CQ)
2. Отфильтровать по дате: 23.04
3. Искать событие `pa_doi`

**Что должно быть видно:**
- Событие `pa_doi` в 14:14 МСК (или близко) — webhook платформы сработал, отдал событие CQ
- Свойство `pa_doi` в карточке лида со ссылкой вида `https://saecuecevicwjkpmaoot.supabase.co/auth/v1/verify?token=...`

**Возможные исходы:**

| Что видно | Что это значит |
|-----------|----------------|
| Событие `pa_doi` 23.04 есть, свойство `pa_doi` есть | Платформа отработала ОК. Проблема дальше — в automation rule CQ |
| События `pa_doi` 23.04 нет | Webhook не дошёл до CQ. Проблема на стороне платформы (но мы зафиксили это сегодня в Sentry — будем видеть в будущем) |
| Событие есть, но свойство `pa_doi` пустое | Баг в нашем `setUserProps` или CQ не сохранил props. Пинг разработчику |

### Шаг 3. Проверить automation rule для `pa_doi`

1. Раздел **«Автоматические сообщения»** / **«Triggered emails»** / **«Automation»**
2. Найти rule с триггером — событие `pa_doi`
3. Открыть его настройки

**Что проверить:**
- ✅ **Статус — активен** (не отключён)
- ✅ **Триггер** — событие `pa_doi`
- ✅ **Шаблон письма** — подключён, рендерится без ошибок, использует переменную `{{pa_doi}}` для CTA-ссылки
- ✅ **Условия / фильтры** — нет ли отрезающего сегмента (например, исключение `подписан на маркетинг = false`, или country/lang фильтр, который мог отрезать bakaresh)
- ✅ **Throttle / лимиты** — не стоит ли «отправлять не чаще 1 раза в день» или подобное

**Если rule отключён или фильтры режут аудиторию** — это и есть причина. Нужно поправить.

### Шаг 4. Проверить историю отправок (Send log) для лида

1. В карточке лида или в разделе **«Сообщения / Письма»** найти историю отправленных писем для bakaresh
2. Отфильтровать по 23.04

**Что искать:**

| Что видно | Что это значит |
|-----------|----------------|
| Письмо отправлено 23.04, статус «Доставлено» | Письмо ушло, проблема у Yandex (фильтр / спам). Юзеру предложить проверить «все папки» в Yandex.Почте, включая «Удалённые» и «Спам». Можно дёрнуть `pa_doi` повторно после warm-up |
| Письмо отправлено, статус «Bounce» / «Hard bounce» | Yandex отверг письмо. Возможно лид в suppression list. Снять suppress, повторить |
| Письмо отправлено, статус «Spam complaint» | Юзер пометил наше письмо спамом раньше. Нужна работа с deliverability на стороне CQ |
| Письма НЕТ в истории | Automation rule не сработал. Проверить шаг 3 |

### Шаг 5. Проверить suppression list / unsubscribe / blacklist

1. Раздел **«Suppression list»** / **«Отписки»** / **«Blacklist»** в CQ
2. Поиск `bakaresh@yandex.ru`

Если найден — снять отметку. Это может объяснить, почему automation rule не отправил письмо.

### Шаг 6. Проверить deliverability-метрики канала за 23.04

1. Раздел **«Аналитика»** / **«Reports»** в CQ
2. Период: 22-23.04
3. Метрики по automation rule `pa_doi`:
   - Отправлено vs Доставлено vs Bounce
   - Spam complaint rate
   - Open rate

Если 23.04 был аномалия (всплеск bounce / низкий delivery rate на Yandex) — это системная проблема с deliverability, не с конкретным юзером.

## Бонус — что мы уже сделали со стороны разработки

Сегодня вечером (27.04) задеплоили observability-фикс: теперь все ошибки CarrotQuest API автоматически попадают в наш Sentry с тегами `area:carrotquest-email` или `area:supabase-email-hook`. До этого силент-дропы (как у bakaresh) терялись в логах Docker без следов. Так что в будущем при подобном тикете — за минуту в Sentry мы увидим, был ли вызван webhook, отвалилась ли CQ, и для какого юзера.

Если хотите — можем настроить Sentry alert rule, чтобы CRM получала пинг в Telegram при такой ошибке быстрее, чем юзер пишет в поддержку.
