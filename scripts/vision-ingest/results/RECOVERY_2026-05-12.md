## R1: Lesson.order migration recovery ‚Äî DONE

- Ran in-place renumber via Supabase Management API (same SQL as commit `6c7bbc2`):
  ```sql
  UPDATE "Lesson" SET "order" = sub.rn FROM (
    SELECT id, ROW_NUMBER() OVER (PARTITION BY "courseId" ORDER BY "order", id) AS rn
    FROM "Lesson"
  ) sub WHERE "Lesson".id = sub.id AND "Lesson"."order" != sub.rn;
  ```
  ‚Üí 257 lessons renumbered (exact match to original migration count).
- Added UNIQUE constraint via Management API:
  ```sql
  ALTER TABLE "Lesson" ADD CONSTRAINT "Lesson_courseId_order_key" UNIQUE ("courseId", "order");
  ```
- Registered migration in `_prisma_migrations` (manual INSERT) so future `prisma migrate deploy` doesn't try re-applying.

Verified: 439 lessons / 439 distinct (`courseId`, `order`) pairs. UNIQUE constraint present. Migration listed in `_prisma_migrations`. Elena's prev/next bug fixed.

Why manual via Management API and not `prisma migrate deploy`? Owner's new safety rule (after incident): no prisma CLI commands against prod DB without extreme care. Manual ALTER + INSERT into `_prisma_migrations` is the same end-state with zero risk of accidental `--accept-data-loss` or unintended schema drift.
